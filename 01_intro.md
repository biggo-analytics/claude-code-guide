# บทนำและพื้นฐาน

## Introduction & Fundamentals

คู่มือภายในสำหรับการใช้งาน Claude Code ซึ่งเป็นเครื่องมือเอเจนต์ (Agent) แบบ CLI ของ Anthropic
เอกสารนี้ครอบคลุมพื้นฐานทั้งหมดที่จำเป็นสำหรับการเริ่มต้นใช้งาน ตั้งแต่การติดตั้ง การตั้งค่า
โหมดใช้งาน คำสั่งพื้นฐาน ไปจนถึงโครงสร้างโปรเจกต์ที่เกี่ยวข้อง

---

## 1.1 Claude Code คืออะไร

### What is Claude Code?

Claude Code เป็นเครื่องมือเอเจนต์ (Agent) สำหรับพัฒนาซอฟต์แวร์ผ่านบรรทัดคำสั่ง (Command Line Interface)
พัฒนาโดย Anthropic ซึ่งเป็นบริษัทผู้สร้างโมเดล (Model) Claude

### หลักการทำงาน

Claude Code ทำงานผ่านเทอร์มินัล (Terminal) ในรูปแบบ agentic coding tool โดยมีลักษณะเด่นดังนี้:

- **ทำงานแบบเอเจนต์ (Agentic):** คิด วางแผน ลงมือทำ และตรวจสอบผลลัพธ์ด้วยตัวเอง
- **ใช้โมเดล Claude เป็นแกนหลัก:** รองรับ Sonnet, Opus และ Haiku
- **อ่าน แก้ไข และสร้างไฟล์ได้เอง:** ไม่จำเป็นต้อง copy-paste โค้ดกลับไปกลับมา
- **รันคำสั่งในเทอร์มินัลได้:** เช่น `git`, `npm`, `go build` ฯลฯ
- **ทำงานข้ามโปรเจกต์ได้:** สามารถอ้างอิงและแก้ไขไฟล์จากหลายโปรเจกต์พร้อมกัน
- **มีระบบสิทธิ์ (Permission System):** ขออนุญาตก่อนดำเนินการที่อาจส่งผลกระทบ

### ความแตกต่างจากเครื่องมืออื่น

| คุณสมบัติ | GitHub Copilot | Cursor | Claude Code |
|-----------|---------------|--------|-------------|
| รูปแบบ | Inline autocomplete | AI-native IDE | CLI-based agent |
| สภาพแวดล้อม | ทำงานใน IDE (VS Code, JetBrains ฯลฯ) | ทำงานในตัว editor ของ Cursor | ทำงานในเทอร์มินัลใดก็ได้ |
| การทำงาน | เติมโค้ดอัตโนมัติ | แก้ไขโค้ดตามคำสั่ง | อ่าน/แก้ไข/สร้างไฟล์ + รันคำสั่ง |
| ขอบเขต | ไฟล์ที่เปิดอยู่ | โปรเจกต์ปัจจุบัน | ข้ามโปรเจกต์ได้ |
| ระบบสิทธิ์ | ไม่มี | จำกัด | มีระบบสิทธิ์เต็มรูปแบบ |
| การทำงานแบบ Agentic | ไม่มี | บางส่วน | เต็มรูปแบบ (คิด วางแผน ทำ ตรวจสอบ) |

### กรณีใช้งาน (Use Cases)

Claude Code เหมาะสำหรับงานหลากหลายรูปแบบ:

1. **สร้างโค้ด (Code Generation):** สร้างฟังก์ชัน, API endpoint, หน้า UI จากคำอธิบาย
2. **แก้บัค (Debugging):** วิเคราะห์ error log และหาทางแก้ไข
3. **รีแฟคเตอร์ (Refactoring):** ปรับโครงสร้างโค้ดให้ดีขึ้นโดยไม่เปลี่ยนพฤติกรรม
4. **รีวิวโค้ด (Code Review):** ตรวจสอบโค้ดและเสนอแนะการปรับปรุง
5. **วิเคราะห์สถาปัตยกรรม (Architecture Analysis):** ศึกษาและทำความเข้าใจ codebase ขนาดใหญ่
6. **แปลงสเปคเป็นโค้ด (Spec-to-Code):** สร้างโค้ดจากเอกสารสเปค (Specification)

## 1.2 การติดตั้งและตั้งค่า

### Installation & Configuration

### ความต้องการของระบบ (System Requirements)

| รายการ | ข้อกำหนด |
|--------|----------|
| ระบบปฏิบัติการ | macOS, Linux, หรือ WSL บน Windows |
| Node.js | เวอร์ชัน 18 ขึ้นไป |
| เทอร์มินัล | เทอร์มินัลใดก็ได้ที่รองรับ ANSI escape codes |
| เครือข่าย | ต้องเชื่อมต่ออินเทอร์เน็ต (สำหรับเรียก API) |

### การติดตั้ง

```bash
# ติดตั้ง Claude Code ผ่าน npm
npm install -g @anthropic-ai/claude-code

# หรือติดตั้งผ่าน script
curl -fsSL https://claude.ai/install.sh | bash

# ตรวจสอบเวอร์ชันที่ติดตั้ง
claude --version

# อัพเดทเป็นเวอร์ชันล่าสุด
claude update
```

> **หมายเหตุ:** หากใช้ macOS อาจต้องอนุญาตสิทธิ์ใน System Preferences > Security & Privacy ก่อนรันครั้งแรก

### การยืนยันตัวตน (Authentication)

```bash
# เปิด Claude Code แล้วเลือก Login
claude

# ระบบจะเปิดเบราว์เซอร์ให้ยืนยันตัวตนผ่าน Anthropic Console
# เมื่อยืนยันสำเร็จ จะกลับมาที่เทอร์มินัลโดยอัตโนมัติ
```

ขั้นตอนการยืนยันตัวตน:
1. รันคำสั่ง `claude` ในเทอร์มินัล
2. เลือก **Login** จากเมนู
3. เบราว์เซอร์จะเปิดหน้า authentication ของ Anthropic
4. ลงชื่อเข้าใช้ด้วยบัญชี Anthropic
5. อนุญาตให้ Claude Code เข้าถึงบัญชี
6. กลับมาที่เทอร์มินัล — พร้อมใช้งาน

### การเลือกโมเดล (Model Selection)

Claude Code รองรับโมเดลหลัก 3 ตัว:

| โมเดล | Model ID | ลักษณะ | เหมาะกับ |
|-------|----------|--------|----------|
| Sonnet | `claude-sonnet-4-6` | เร็ว ประหยัด สมดุล | งานทั่วไป (default) |
| Opus | `claude-opus-4-6` | ฉลาดที่สุด แพงกว่า | งานซับซ้อน วิเคราะห์เชิงลึก |
| Haiku | `claude-haiku-4-5` | เร็วที่สุด ถูกที่สุด | งานง่ายๆ ที่ต้องการความเร็ว |

วิธีเลือกโมเดล:

```bash
# เลือกโมเดลตอนเปิด session
claude --model sonnet
claude --model opus
claude --model haiku

# เปลี่ยนโมเดลระหว่าง session ด้วย slash command
/model
```

### การตั้งค่า API Key

สำหรับการใช้งานผ่าน API Key โดยตรง:

```bash
# ตั้งค่า API Key ผ่าน environment variable
export ANTHROPIC_API_KEY="sk-ant-xxxxx"

# เพิ่มใน shell profile เพื่อใช้งานถาวร
echo 'export ANTHROPIC_API_KEY="sk-ant-xxxxx"' >> ~/.zshrc
source ~/.zshrc
```

> **คำเตือน:** อย่า commit ไฟล์ที่มี API Key ลง Git เด็ดขาด ควรใช้ `.env` และเพิ่ม `.env` ใน `.gitignore`

### การใช้งานผ่าน Cloud Provider

```bash
# ใช้งานผ่าน AWS Bedrock
export CLAUDE_CODE_USE_BEDROCK=1
export AWS_REGION="us-east-1"

# ใช้งานผ่าน Google Vertex AI
export CLAUDE_CODE_USE_VERTEX=1
export CLOUD_ML_REGION="us-central1"
export ANTHROPIC_VERTEX_PROJECT_ID="your-project-id"
```

> **เคล็ดลับ:** การใช้งานผ่าน Bedrock หรือ Vertex เหมาะสำหรับองค์กรที่มีข้อกำหนดด้านความปลอดภัยและการเรียกเก็บเงินผ่าน cloud provider โดยตรง

## 1.3 โหมดใช้งาน

### Usage Modes

Claude Code มีโหมดใช้งานหลัก 2 โหมด แต่ละโหมดเหมาะกับสถานการณ์ที่แตกต่างกัน

### โหมดโต้ตอบ (Interactive Mode)

โหมดโต้ตอบเป็นโหมดหลักที่ใช้บ่อยที่สุด เปิดเซสชัน (Session) แบบต่อเนื่อง สามารถสนทนาโต้ตอบกับ Claude ได้หลายรอบ

```bash
# เปิด session ใหม่
claude

# ต่อ session ล่าสุด
claude -c

# เลือก session เก่าจากรายการ
claude --resume

# เปิดพร้อมระบุ model
claude --model opus
```

**ขั้นตอนการทำงานในโหมดโต้ตอบ:**

1. พิมพ์คำถามหรือคำสั่งเป็นภาษาธรรมชาติ
2. Claude วิเคราะห์คำสั่งและวางแผนการทำงาน
3. Claude ขออนุญาตก่อนใช้เครื่องมือ (Tool) ที่อาจส่งผลกระทบ
4. Claude ดำเนินการ (อ่านไฟล์, แก้ไขโค้ด, รันคำสั่ง)
5. แสดงผลลัพธ์และสรุปสิ่งที่ทำ
6. รอคำสั่งถัดไป

**ตัวอย่างการใช้งาน:**

```
You: อธิบายโครงสร้างของโปรเจกต์นี้ให้หน่อย
Claude: [อ่านไฟล์ต่างๆ แล้วสรุปโครงสร้าง]

You: สร้าง API endpoint สำหรับ CRUD ของ users
Claude: [สร้างไฟล์ เขียนโค้ด ทดสอบ]
```

> **เคล็ดลับ:** ใช้ `claude -c` เพื่อกลับมาต่อ session ล่าสุด ไม่ต้องอธิบายบริบท (Context) ใหม่ทั้งหมด

### โหมดไม่โต้ตอบ (Non-Interactive Mode)

โหมดไม่โต้ตอบเหมาะสำหรับงานที่ต้องการรันครั้งเดียวจบ หรือใช้ร่วมกับสคริปต์ (Script) และระบบ CI/CD

```bash
# ส่งคำสั่งเดียวด้วย flag -p (print mode)
claude -p "สร้าง unit test สำหรับ handler/user.go"

# จำกัดจำนวนรอบการทำงาน
claude -p "fix the failing test" --max-turns 5

# ใช้กับท่อ/ไปป์ (Pipe) เพื่อส่ง input
cat error.log | claude -p "วิเคราะห์ error นี้และเสนอวิธีแก้ไข"

# ส่ง output เป็น JSON
claude -p "list all TODO comments" --output-format json

# ส่ง output เป็น text ธรรมดา (ไม่มี markdown)
claude -p "สรุปการเปลี่ยนแปลงใน git log 5 commits ล่าสุด" --output-format text
```

**กรณีใช้งานโหมดไม่โต้ตอบ:**

| กรณี | ตัวอย่างคำสั่ง |
|------|---------------|
| สร้างโค้ดจากคำอธิบาย | `claude -p "สร้าง middleware สำหรับ rate limiting"` |
| วิเคราะห์ error | `cat error.log \| claude -p "วิเคราะห์และแก้ไข"` |
| ทำงานใน CI/CD | `claude -p "run tests and fix failures" --max-turns 10` |
| สร้างเอกสาร | `claude -p "สร้าง API documentation จาก handler/*.go"` |
| แปลงรูปแบบ | `claude -p "แปลง config.yaml เป็น config.json"` |

> **หมายเหตุ:** ในโหมดไม่โต้ตอบ (`-p`) Claude จะยอมรับสิทธิ์อ่านไฟล์โดยอัตโนมัติ แต่จะไม่แก้ไขหรือรันคำสั่งที่อาจเป็นอันตรายโดยไม่ได้รับอนุญาต

## 1.4 คำสั่งสแลชพื้นฐาน

### Basic Slash Commands

คำสั่งสแลช (Slash Command) คือคำสั่งพิเศษที่ใช้ในโหมดโต้ตอบ เริ่มต้นด้วยเครื่องหมาย `/`

| คำสั่ง | คำอธิบาย | ตัวอย่างการใช้งาน |
|--------|----------|------------------|
| `/help` | แสดงรายการคำสั่งทั้งหมดที่ใช้ได้ | พิมพ์ `/help` เมื่อไม่แน่ใจว่ามีคำสั่งอะไรบ้าง |
| `/clear` | ล้างบริบท (Context) ทั้งหมดในการสนทนา | ใช้เมื่อต้องการเริ่มต้นใหม่โดยไม่ต้องเปิด session ใหม่ |
| `/compact` | บีบอัดบริบทเพื่อประหยัดโทเค็น (Token) | ใช้เมื่อ session ยาวเกินไปและเริ่มใช้โทเค็นมาก |
| `/context` | แสดงรายการไฟล์ที่อยู่ในบริบทปัจจุบัน | ใช้ตรวจสอบว่า Claude กำลังอ้างอิงไฟล์ใดบ้าง |
| `/cost` | แสดงค่าใช้จ่ายโทเค็นของ session ปัจจุบัน | ใช้ติดตามการใช้งานและงบประมาณ |
| `/model` | เปลี่ยนโมเดลระหว่าง session | สลับจาก Sonnet เป็น Opus สำหรับงานยาก |
| `/rename` | เปลี่ยนชื่อ session | ตั้งชื่อให้จำง่าย เช่น "fix-auth-bug" |
| `/memory` | แก้ไข MEMORY.md สำหรับจดจำข้อมูลข้าม session | บันทึกข้อตกลง coding style ของทีม |
| `/init` | สร้าง CLAUDE.md ให้โปรเจกต์ | ใช้ครั้งแรกเมื่อเริ่มใช้ Claude Code กับโปรเจกต์ใหม่ |
| `/review` | รีวิว (Review) โค้ดใน session ปัจจุบัน | ขอให้ Claude ตรวจสอบโค้ดที่เพิ่งเขียน |

### รายละเอียดคำสั่งที่สำคัญ

**`/compact` — บีบอัดบริบท**

เมื่อ session ดำเนินไปนานๆ จำนวนโทเค็นจะเพิ่มขึ้นเรื่อยๆ ทำให้:
- ค่าใช้จ่ายสูงขึ้น
- การตอบสนองช้าลง
- อาจเกินขีดจำกัดของหน้าต่างบริบท (Context Window)

ใช้ `/compact` เพื่อให้ Claude สรุปบริบทเดิมแบบกระชับ ลดจำนวนโทเค็นโดยยังคงข้อมูลสำคัญไว้

**`/init` — สร้าง CLAUDE.md**

คำสั่งนี้จะให้ Claude วิเคราะห์โปรเจกต์และสร้างไฟล์ `CLAUDE.md` ที่มีข้อมูล:
- ภาษาและเฟรมเวิร์คที่ใช้
- คำสั่ง build/test/lint
- โครงสร้างโปรเจกต์
- coding conventions ของทีม

> **เคล็ดลับ:** รัน `/init` ทุกครั้งเมื่อเริ่มใช้ Claude Code กับโปรเจกต์ใหม่ เพราะ CLAUDE.md ช่วยให้ Claude เข้าใจโปรเจกต์ได้ดีขึ้นอย่างมาก

## 1.5 คีย์ลัด

### Keyboard Shortcuts

คีย์ลัดช่วยให้ทำงานกับ Claude Code ได้เร็วขึ้นโดยไม่ต้องพิมพ์คำสั่ง

| คีย์ลัด | การทำงาน | สถานการณ์ที่ใช้ |
|---------|----------|----------------|
| `Esc` | ยกเลิกคำสั่งปัจจุบัน / หยุด generation | Claude กำลังสร้างโค้ดที่ไม่ตรงตามต้องการ ต้องการหยุดทันที |
| `Esc` + `Esc` | ยกเลิกและย้อนกลับ (rewind) | ยกเลิกการทำงานล่าสุดและกลับไปยังจุดก่อนหน้า |
| `Shift+Tab` | สลับเป็นโหมดแผน (Plan Mode) | ต้องการให้ Claude วางแผนก่อนลงมือทำ ไม่ต้องการให้แก้ไขไฟล์ทันที |
| `Ctrl+B` | สลับ bookmark | ทำเครื่องหมายจุดสำคัญในการสนทนาเพื่อกลับมาดูภายหลัง |
| `Tab` | ยอมรับ suggestion (ในบาง context) | เมื่อ Claude เสนอคำสั่งหรือเส้นทางไฟล์ กด Tab เพื่อยอมรับ |

### รายละเอียดการใช้งานคีย์ลัด

**`Esc` — ยกเลิกคำสั่ง**

กด `Esc` หนึ่งครั้งเพื่อหยุดการทำงานปัจจุบันของ Claude ทันที ใช้เมื่อ:
- Claude กำลังสร้างโค้ดที่ไม่ตรงตามต้องการ
- ต้องการเปลี่ยนคำสั่งกลางคัน
- Claude กำลังอ่านไฟล์ที่ไม่เกี่ยวข้อง

**`Esc` + `Esc` — ยกเลิกและย้อนกลับ**

กด `Esc` สองครั้งติดต่อกันเพื่อยกเลิกการทำงานล่าสุดและกลับไปยังสถานะก่อนหน้า ใช้เมื่อ:
- Claude แก้ไขไฟล์ผิดและต้องการย้อนกลับ
- ต้องการลองวิธีอื่นตั้งแต่ต้น

**`Shift+Tab` — โหมดแผน (Plan Mode)**

สลับเข้าสู่โหมดแผนเพื่อให้ Claude วางแผนการทำงานก่อนลงมือทำ ในโหมดนี้ Claude จะ:
- อธิบายขั้นตอนที่จะทำ
- ระบุไฟล์ที่จะแก้ไข
- ไม่แก้ไขไฟล์จริงจนกว่าจะได้รับอนุญาต

> **เคล็ดลับ:** ใช้โหมดแผนเสมอสำหรับงานที่ซับซ้อนหรืองานที่มีผลกระทบกว้าง เช่น การรีแฟคเตอร์ (Refactor) ขนาดใหญ่ หรือการเปลี่ยนสถาปัตยกรรม

## 1.6 โครงสร้างโปรเจกต์ที่เกี่ยวข้อง

### Project Structure

Claude Code ใช้ไฟล์และไดเรกทอรีหลายรายการในการเก็บการตั้งค่าและคำแนะนำ

### ภาพรวมโครงสร้าง

```
project-root/
├── CLAUDE.md                    # คำแนะนำหลักของโปรเจกต์
├── MEMORY.md                    # หน่วยความจำถาวรข้าม session
├── .claude/
│   ├── settings.json            # ตั้งค่าโปรเจกต์ (แชร์ผ่าน git)
│   ├── settings.local.json      # ตั้งค่าเฉพาะเครื่อง (gitignored)
│   ├── rules/                   # กฎเพิ่มเติมสำหรับ Claude
│   │   ├── coding-style.md
│   │   └── review-checklist.md
│   └── skills/                  # ทักษะที่กำหนดเอง
│       ├── deploy.md
│       └── migration.md
└── src/
    └── ...
```

### รายละเอียดแต่ละไฟล์

**`CLAUDE.md` — สมองของโปรเจกต์**

ไฟล์ที่สำคัญที่สุดสำหรับ Claude Code ทำหน้าที่เป็นคำแนะนำหลักที่ Claude จะอ่านทุกครั้งที่เริ่ม session ควรมีข้อมูล:
- คำสั่ง build, test, lint ที่ใช้ในโปรเจกต์
- โครงสร้างไดเรกทอรีและหน้าที่ของแต่ละส่วน
- coding conventions และ style guide
- ข้อห้ามและข้อควรระวัง

> **หมายเหตุ:** สร้างไฟล์นี้ได้ง่ายด้วยคำสั่ง `/init` ภายใน session

**`.claude/settings.json` — ตั้งค่าโปรเจกต์ (แชร์ได้)**

ไฟล์นี้เก็บการตั้งค่าที่แชร์กับทีมผ่าน Git เช่น:
- คำสั่งที่อนุญาตให้รันโดยไม่ต้องขออนุญาต
- เครื่องมือ MCP (Model Context Protocol) ที่ใช้
- การตั้งค่าสิทธิ์เริ่มต้น

```json
{
  "permissions": {
    "allow": [
      "Bash(go build ./...)",
      "Bash(go test ./...)",
      "Bash(npm run lint)"
    ],
    "deny": [
      "Bash(rm -rf /)"
    ]
  }
}
```

**`.claude/settings.local.json` — ตั้งค่าเฉพาะเครื่อง**

เหมือน `settings.json` แต่ไม่ถูก commit ลง Git เหมาะสำหรับ:
- การตั้งค่าเฉพาะตัวนักพัฒนา
- เส้นทางไฟล์ที่แตกต่างกันในแต่ละเครื่อง
- API keys หรือ credentials เฉพาะคน

**`.claude/rules/` — กฎเพิ่มเติม**

ไดเรกทอรีสำหรับเก็บกฎหรือคำแนะนำเพิ่มเติมในรูปแบบไฟล์ markdown แต่ละไฟล์สามารถกำหนดกฎเฉพาะเรื่อง เช่น:
- `coding-style.md` — กฎการเขียนโค้ด
- `review-checklist.md` — รายการตรวจสอบสำหรับรีวิวโค้ด
- `api-conventions.md` — มาตรฐานการออกแบบ API

**`.claude/skills/` — ทักษะที่กำหนดเอง**

ไดเรกทอรีสำหรับเก็บทักษะ (Skill) ที่สร้างขึ้นเอง เป็นไฟล์ markdown ที่อธิบายวิธีทำงานเฉพาะทาง เช่น:
- `deploy.md` — ขั้นตอนการดีพลอย (Deploy)
- `migration.md` — ขั้นตอนการทำไมเกรชัน (Migration)
- `create-api.md` — ขั้นตอนการสร้าง API ใหม่

**`MEMORY.md` — หน่วยความจำถาวร**

ไฟล์สำหรับเก็บข้อมูลที่ Claude ควรจำข้าม session เช่น:
- การตัดสินใจทางสถาปัตยกรรมที่สำคัญ
- ข้อตกลงของทีม
- ปัญหาที่เจอบ่อยและวิธีแก้ไข

แก้ไขได้ด้วยคำสั่ง `/memory` ภายใน session

> **เคล็ดลับ:** commit ไฟล์ `CLAUDE.md`, `.claude/settings.json`, `.claude/rules/` และ `.claude/skills/` ลง Git เพื่อให้ทุกคนในทีมได้ประโยชน์ร่วมกัน

## 1.7 อภิธานศัพท์ ไทย-อังกฤษ

### Thai-English Glossary

ตารางศัพท์เทคนิคที่ใช้ตลอดคู่มือนี้ เรียงตามลำดับตัวอักษรภาษาไทย

| ศัพท์ภาษาไทย | English Term | คำอธิบาย |
|-------------|-------------|----------|
| เครื่องมือ | Tool | ฟังก์ชันที่ Claude เรียกใช้ เช่น Read, Edit, Bash, Grep |
| คำสั่งสแลช | Slash Command | คำสั่งพิเศษในโหมดโต้ตอบ เริ่มด้วย `/` เช่น `/help`, `/compact` |
| ซับเอเจนต์ | Subagent | เอเจนต์ย่อยที่ถูกเรียกใช้โดยเอเจนต์หลักเพื่อทำงานเฉพาะทาง |
| ดีพลอย | Deploy | การนำซอฟต์แวร์ไปติดตั้งหรือเผยแพร่บนเซิร์ฟเวอร์จริง |
| โทเค็น | Token | หน่วยวัดข้อมูลที่โมเดล AI ประมวลผล ใช้คำนวณค่าใช้จ่าย |
| ท่อ/ไปป์ | Pipe | การส่ง output ของคำสั่งหนึ่งเป็น input ของอีกคำสั่ง ด้วยเครื่องหมาย `\|` |
| บริบท | Context | ข้อมูลทั้งหมดที่ Claude ใช้อ้างอิงในการสนทนาปัจจุบัน |
| โมเดล | Model | แบบจำลอง AI ที่ใช้ประมวลผล เช่น Sonnet, Opus, Haiku |
| โหมดโต้ตอบ | Interactive Mode | โหมดที่สนทนากับ Claude แบบต่อเนื่อง |
| โหมดไม่โต้ตอบ | Non-Interactive Mode | โหมดที่ส่งคำสั่งเดียวด้วย `-p` flag |
| โหมดแผน | Plan Mode | โหมดที่ Claude วางแผนก่อนลงมือทำ เปิดด้วย `Shift+Tab` |
| พรอมต์ | Prompt | ข้อความคำสั่งหรือคำถามที่ส่งให้ AI |
| เอเจนต์ | Agent | โปรแกรม AI ที่ทำงานอัตโนมัติ วางแผนและใช้เครื่องมือได้ |
| เอ็นด์พอยต์ | Endpoint | จุดเชื่อมต่อ API ที่รับ request เช่น `GET /api/users` |
| แซนด์บ็อกซ์ | Sandbox | สภาพแวดล้อมที่แยกออกมาสำหรับทดสอบอย่างปลอดภัย |
| สคีมา | Schema | โครงสร้างข้อมูลที่กำหนดไว้ เช่น database schema, JSON schema |
| สคริปต์ | Script | ชุดคำสั่งที่เขียนไว้ล่วงหน้าเพื่อรันอัตโนมัติ |
| สเปค | Specification | เอกสารข้อกำหนดที่อธิบายพฤติกรรมที่ต้องการของระบบ |
| เซสชัน | Session | ช่วงเวลาการสนทนาหนึ่งครั้งกับ Claude |
| รีแฟคเตอร์ | Refactor | การปรับโครงสร้างโค้ดให้ดีขึ้นโดยไม่เปลี่ยนพฤติกรรม |
| รีวิว | Review | การตรวจสอบและให้ความเห็นเกี่ยวกับโค้ด |
| ระบบสิทธิ์ | Permission System | กลไกที่ควบคุมว่า Claude ทำอะไรได้บ้างโดยไม่ต้องขออนุญาต |
| เวิร์คทรี | Worktree | สำเนาของ Git repository ที่ทำงานแยกกันได้ ใช้สำหรับ parallel tasks |
| หน้าต่างบริบท | Context Window | ขนาดสูงสุดของข้อมูลที่โมเดล AI สามารถประมวลผลได้ในครั้งเดียว |
| ฮุค | Hook | สคริปต์ที่ทำงานอัตโนมัติเมื่อเกิดเหตุการณ์บางอย่าง เช่น pre-commit hook |
| ไมเกรชัน | Migration | การเปลี่ยนแปลงโครงสร้างฐานข้อมูลอย่างเป็นระบบ |
| MCP | Model Context Protocol | โปรโตคอลสำหรับเชื่อมต่อ Claude กับเครื่องมือภายนอก |
| CLAUDE.md | CLAUDE.md | ไฟล์คำแนะนำหลักของโปรเจกต์ที่ Claude อ่านทุกครั้ง |
| CI/CD | Continuous Integration / Continuous Deployment | กระบวนการอัตโนมัติสำหรับ build, test, และ deploy ซอฟต์แวร์ |
| API Key | API Key | กุญแจสำหรับยืนยันตัวตนเมื่อเรียกใช้ API |
